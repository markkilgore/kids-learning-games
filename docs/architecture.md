# Monorepo architecture and migration guide

## 1. Initial codebase assessment

The starting repository was a compact SwiftUI iPad application generated by XcodeGen. It had one application target and one unit-test target. Its strengths were a clear 10-shark JSON catalog, straightforward navigation, encouraging question feedback, two reading modes, bundled/TTS narration, simple SwiftData progress, and app-specific SpriteKit scenes. The main architectural coupling points were global `ContentStore.shared`, `Bundle.main`, Shark-named state fields and destinations, app-local narration/read-along code, and an implicit default persistence location.

The characterized Shark baseline is:

- 10 shark encounters, 50 explore topics, 30 picture questions, and 25 vocabulary entries.
- Three initial unlocks followed by thresholds at 1, 3, and 6 completed sharks.
- Discover/topic narration in Early Reader and Story modes.
- Mission → three questions → shark page and vocabulary reward flow.
- Ocean map, Shark Book, Ocean Words, parent gate/settings, and resume-current-encounter behavior.
- Local completion, topic, vocabulary, reading-mode, and current-encounter persistence.

Tests were added around catalog scope, narration completeness, question validity, unlock thresholds, mission rewards, save/load, and configuration identity before relying on the new boundary.

## 2. Migration plan

The migration is intentionally incremental rather than a rewrite:

1. Characterize and test the Shark content, unlock, reward, and persistence behavior.
2. Move files under `apps/shark-explorer` without changing the Shark-specific models or screens.
3. Extract only configuration value types, namespaced persistence, narration, and read-along UI—systems with two consumers.
4. Give Shark an explicit configuration and move its small progress record into the namespaced store.
5. Add Cat Math as a new shell with its own content, navigation, math challenge factory, progression, and journal.
6. Generate two application targets and verify Debug/Release products, test suites, identifiers, and resource isolation.

Future extractions should follow the “second real consumer” rule. For example, the Cat journal and Shark books are intentionally separate today. A shared journal package becomes appropriate only after their common operations are understood. The same applies to activity generation, adaptive learning, and progression.

## 3. Target repository structure

```text
apps/
  shark-explorer/
    Config/                 # independent Debug/Release settings
    Resources/              # shark catalog, art, audio, icon, splash
    Sources/                # ocean UI, Shark models/content/scenes, shell
    Tests/
  cat-math/
    Config/                 # independent Debug/Release settings
    Resources/              # cat content metadata, icon, splash
    Sources/                # House/Garden, cats, math, journal, shell
    Tests/
packages/
  learning-engine/          # app configuration and reading value types
  persistence/              # validated namespaces and Codable local store
  narration/                # bundled audio/TTS coordinator
  ui-components/            # read-along, word layout, theme color utility
docs/
tools/
project.yml                 # XcodeGen target graph
```

The directory names `activity-engine`, `progression`, `journal`, and `adaptive-learning` are reserved for later, but empty or single-app packages were not created. App-specific world maps, dialogue, activities, rewards, content models, and visual treatments remain with their app.

## 4. Application configuration contract

`LearningAppConfiguration` is a pure value contract. Each app owns one complete instance and passes its parts to the shared services it uses. The contract includes:

| Contract field | Purpose |
| --- | --- |
| `identity` | application ID, display name, child-facing name, icon, splash |
| `theme` | background, primary, secondary, success, and surface tokens |
| `contentPack` | stable pack ID, schema version, bundled resource name |
| `locations` | available app-owned places |
| `characters` | available creatures/characters and their location IDs |
| `enabledSkills` | stable skill IDs and child/parent-facing names |
| `narration` | language, voice rates, pitch, optional audio manifest |
| `rewards` | app-specific singular/plural vocabulary and celebration copy |
| `progression` | model name, points per success, unlock thresholds |
| `storageNamespace` | validated, unique persistence scope |
| `featureFlags` | named capabilities exposed by that app |

`LearningBehaviorHooks<State>` is the injection seam for state-dependent rules such as whether a location can open and how rewards are labeled. It uses closures supplied by an app; shared code does not switch on Shark/Cat themes. Visual composition remains an app responsibility, so Cat can have House/Garden and a friendship journal while Shark keeps its ocean map and books.

Release identity is enforced twice: in the app configuration for runtime systems and in `project.yml`/the app’s `.xcconfig` for the compiled product. Tests pin the expected values.

## 5. Persistence isolation

Isolation has two layers:

1. iOS gives each bundle identifier a separate application sandbox.
2. `NamespacedProgressStore` writes inside `Application Support/<storageNamespace>/progress.json` and rejects path-like namespaces.

Current locations are:

```text
Application Support/com.mkilgore.SharkExplorer.henry/progress.json
Application Support/com.mkilgore.CatMathAdventure.kate/progress.json
```

Resetting one namespace cannot remove the other. The shared persistence test writes different values to both stores, resets Shark, and confirms Cat remains unchanged.

The prototype Shark target used a SwiftData `PlayerProgressRecord`. When the namespaced JSON store does not yet exist, Shark Explorer checks that legacy store and copies any existing completion, vocabulary, topic, encounter, and reading-mode state into the explicit Shark namespace. Existing namespaced progress always takes precedence, and the legacy store is retained rather than becoming an ongoing persistence dependency.

## 6. Build and run commands

Generate:

```sh
xcodegen generate --spec project.yml
```

Build independently:

```sh
xcodebuild -project SharkExplorer.xcodeproj -scheme SharkExplorer \
  -configuration Debug -destination 'generic/platform=iOS Simulator' build

xcodebuild -project SharkExplorer.xcodeproj -scheme CatMathAdventure \
  -configuration Debug -destination 'generic/platform=iOS Simulator' build
```

Release without signing (CI/local verification):

```sh
xcodebuild -project SharkExplorer.xcodeproj -scheme SharkExplorer \
  -configuration Release -destination 'generic/platform=iOS' \
  CODE_SIGNING_ALLOWED=NO build

xcodebuild -project SharkExplorer.xcodeproj -scheme CatMathAdventure \
  -configuration Release -destination 'generic/platform=iOS' \
  CODE_SIGNING_ALLOWED=NO build
```

Install/run from Xcode by choosing the desired scheme and iPad destination. The schemes never select content at runtime; they compile different `@main` apps and different resource phases.

## 7. Commit-sized implementation stages

The supplied source directory did not contain a `.git` repository, so no commits were fabricated. These are the recommended atomic commits if history is initialized or the changes are applied to a tracked upstream repository:

1. `test(shark): characterize catalog unlock rewards and progress flows`
2. `chore(repo): move Shark Explorer under apps/shark-explorer`
3. `feat(core): add app configuration and namespaced persistence contracts`
4. `refactor(audio): extract narration and read-along packages`
5. `refactor(shark): consume shared services in an isolated namespace`
6. `feat(cat): add Cat Math House Garden activities and journal`
7. `build(ios): add separate targets assets schemes and release configs`
8. `test(repo): verify both products build and resources remain isolated`

Each stage should compile; stages 1 and 5 should run Shark regression tests, and stages 6–8 should run both suites.

## 8. Independence and build tests

- `SharkExplorerTests`: catalog scope, reading content, picture questions, guided unlocks, mission rewards/save-load, and identity.
- `CatMathAdventureTests`: two locations, 10 cats, all six requested skills, challenge answer/hint validity, friendship save-load/journal, and identity.
- `LearningPersistenceTests`: namespace validation, separate file paths/values, and reset isolation.
- `tools/verify-monorepo.sh`: Release-builds both schemes, asserts the compiled bundle identifiers/display names, and checks that Cat resources are absent from Shark and Shark catalog/art are absent from Cat.

The target dependency graph is one-way: each app depends on shared frameworks; neither app target depends on the other.
